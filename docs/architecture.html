A coding agent in the spirit of Claude Code — with per-turn cost routing, skills as a first-class primitive, git-style replay and branching of every turn, a live implementation-notes artifact, browser-use as a real tool, and MCP built into the same registry as the native tool surface. Designed end-to-end before code, then built.

§ 01
Six things no shipping coding agent has together
D1
Per-turn cost routing
Every turn picks its own model from a policy. Plan with Haiku, edit with Sonnet, debug with Opus. Same canonical messages, different adapter.
D2
Skills first-class
Loaded like tools, scoped per turn, retrieval-shaped. The system prompt stays thin; expertise is composable on disk.
D3
Replay & branch
Every turn is a content-addressed object. Git for agent sessions: walk, fork, A/B prompts, resume after crash by replay.
D4
Live notes artifact
A black-and-white HTML implementation note rendered at runtime, not just teaching-time. The agent works and the artifact updates.
D5
Browser as a tool
browser-use, one long-lived BrowserSession per tern session. Sub-agent contract — call agent.run() once, surface a structured result.
D6
MCP built-in
Day-one MCP client. Remote tools register into the same registry as native tools, with the same permission gate and the same ToolResult shape.

§ 04
What we lift, what we change

Five invariants we preserve from prior art

  • i. Turn loop = async generator; tool execution = sub-generator. State replaced, not mutated.
  • ii. Tools are uniform: name + input_schema + read_only + concurrency_safe + call. Permissions filtered at registry, enforced at call site.
  • iii. Transcripts append-only JSONL at ~/.tern/projects/<sanitized>/<uuid>.jsonl. Resumable by replay.
  • iv. Reflection retry loop: parse-error / lint-error / test-failure flows back as next user turn, capped by max_reflections.
  • v. Few-shot exemplar in the system prompt that uses the exact edit format on a toy task. Compliance follows from showing.

Three things we deliberately change

  • a. Drop ink + yoga + React. Use textual. Python-native reactive widget tree, CSS theming, mature. Bonus: HTML inspector feeds D4 notes artifact.
  • b. Replace build-time feature flags with runtime plugins + skills. Disk-discovered at startup, not compiled in. Keeps the binary small without sacrificing modularity.
  • c. Provider abstraction owned by the cost router, not by litellm directly. Per-turn model pick. Browser-use and MCP plug into the same registry, not special-cased subsystems.
§ 05
Build order

Phase 1 · skeleton

  • 1. M11 first — observability is bedrock; everything else needs spans to debug and cost-route
  • 2. M0 + M1 + M4 — turn loop + canonical messages + one provider, end-to-end one-shot
  • 3. M5 + M2 — a couple tools and the textual TUI; the agent becomes interactive

Phase 2 · differentiators

  • 4. M7 — session + replay + branch BEFORE M6/M8 (replay needs the object store first)
  • 5. M6 + M8 — skills + live notes artifact; the system starts feeling unique
  • 6. M9 + M10 — browser and MCP; tool surface lights up

Phase 3 · hardening

  • 7. M12 + M13 formalized — timeouts, breakers, redaction, policy file
  • 8. M14 polish — slash commands, keybindings, --print mode
  • 9. Walkthrough authored from .scratch/walkthrough-notes/, captured during 1–8
§ 06
Things this design forbids
  • × The LLM Fallacy. Generated code that looks fine and runs but lacks failure handling, dependency order, or observability. Reject in PR.
  • × Observability as afterthought. M11 ships in Phase 1, not last.
  • × A BaseProviderAdapter with shared logic. Adapters are siblings; share only the canonical type.
  • × Global state in core. Pass session, store, adapter explicitly.
  • × Skills hardcoded into core. Disk-discovered at runtime; never imported.
  • × LLM-generated commit messages. Deterministic tern: <tool> on <files> · turn <id>.
